# v5.2 Implementation Plan — anime id mapping, MAL age ratings, permanent NSFW blacklist

**Status:** Planned, not started. Follows `docs/v5-plan.md`.
**Target release:** `v5.2.0-beta` — `:latest` stays pinned at v2.6.1.

## Why

The v5 age gate is a single LLM judgement. It works, but it has two known
weaknesses, both visible in Ciara's rebuilds:

1. **It over-vetoes gentle anime.** Frieren: Beyond Journey's End and The
   Apothecary Diaries were dropped at 14. Neither is remotely borderline.
2. **It vetoes everything it doesn't know.** Every 2026 premiere on the Anime
   TV-14 list was dropped, because "err on the side of exclusion" fires on any
   title past the model's knowledge cutoff. For a *trending* list that's a
   large slice.

Anime is where both bite hardest, and anime is the one category with an
excellent, near-complete, machine-readable age classification: MyAnimeList's
`rating` field. This stage uses data where data exists and keeps the LLM for
everything else.

## What the research settled

Reviewed [cedya77/aiometadata](https://github.com/cedya77/aiometadata), which
solves anime id mapping thoroughly and solves age gating not at all.

**Copy their id mapping.** [Fribb's anime-lists](https://github.com/Fribb/anime-lists)
`anime-list-full.json` — verified live at **7.5 MB with a working ETag**. Maps
`mal_id ↔ anilist_id ↔ kitsu_id ↔ anidb_id ↔ imdb_id ↔ themoviedb_id ↔
tvdb_id ↔ simkl_id`, plus type.

**Do NOT expect an age system from them.** AIOMetadata has no parental gating.
AniList `isAdult` and Kitsu `filter[ageRating]` are used only as NSFW search
filters — "hide porn", not "suitable for a 13-year-old". We build the age
layer ourselves.

**The Fribb file carries ids only, no ratings.** Mapping is free and offline;
the rating still needs one Jikan/Kitsu call per title. Ratings are static, so
that is a one-time cost per title, not per rebuild.

---

## The two rules (they point in opposite directions — deliberately)

This is the heart of the design. Getting these backwards rebuilds the Common
Sense Media failure with a better database behind it.

| | Rule | Rationale |
|---|---|---|
| **NSFW blacklist** | **ANY** positive adult signal blocks, permanently, on every profile including adults. | A hentai flag is a positive assertion. Presence is terminal. |
| **Age rating** | Only a **known** rating **above** the limit drops a title. Absent rating → falls through to the LLM. **Never** drop for being unrated. | "No rating" is not "too old". CSM dropped unrated titles, anime is largely unrated, and kids catalogs came back empty. |

So: fail-**closed** on an explicit adult flag, fail-**through** on a missing
age rating.

## Permanent NSFW / porn / hentai blacklist

Not tied to `age_limit`. Not a filter a profile can relax. Blocked for every
profile, on every surface.

Signals, any one of which blocks:

- MAL `rating` = `rx` (Hentai)
- AniList `isAdult: true`
- Kitsu `ageRating` = `R18`, or `nsfw: true`
- Our own permanent id denylist (a small local list, for anything the above
  miss — appended to, never subtracted from without a deliberate commit)

`r+` (MAL "Mild Nudity") is **not** on the permanent blacklist — it isn't
pornography — but it is blocked for any profile with an age limit set. Adults
keep it. Flagging this as the one judgement call in the table above.

**Applies to `meta` as well.** v5 left `meta` ungated because an LLM call in
front of every title open was unacceptable latency. That reasoning does not
apply here: the blacklist is a local map lookup against a cached rating, with
no network and no model. It is cheap enough to run everywhere, so it runs
everywhere — lists, extras, search, and meta.

TMDB's `include_adult: false` stays on as an independent upstream layer. Two
cheap independent filters beat one.

## Pipeline

```
title (tt / tmdb id)
  │
  ├─ Fribb map lookup ────────────► not found → NOT anime → existing LLM path
  │                                             (unchanged)
  ├─ mal_id → cached MAL rating
  │
  ├─ [C] NSFW blacklist ──────────► rx / isAdult / R18 → DROP, always, all ages
  │
  ├─ [D] anime age compare
  │        rating known & > limit → DROP
  │        rating known & ≤ limit → keep, marked "rated by MAL"
  │        rating absent          → keep, fall through to [E]
  │
  └─ [E] LLM stage 2 safeguard ───► runs on everything that survives,
           remove-only, fail-closed, unchanged from v5
```

**Stage E is retained in full.** MAL narrows and informs; it never replaces the
safeguard. Nothing reaches a child on a data lookup alone. Titles MAL has
already cleared still go to the LLM, and the LLM's verdict — being remove-only
— can still veto them.

MAL's verdict IS passed to the LLM as evidence (`certification: "PG-13"`),
which should also reduce the Frieren-class over-vetoes: the model is currently
guessing at a rating that we could simply hand it.

## Deliverable 1 — anime id map + ratings

- `src/services/animeMap.js`: fetch, ETag-conditional refresh (HEAD first,
  compare stored ETag, skip download if unchanged; 24 h interval), disk cache,
  build indices at load.
- **Slim the index.** Keep only `{ mal_id, kitsu_id, anilist_id, type }` keyed
  by imdb id and tmdb id; discard the rest of each record. No reason to hold
  7.5 MB of parsed objects resident.
- **Franchise arrays.** `imdb_id` and `themoviedb_id` are sometimes ARRAYS —
  Fribb collapses multi-part franchises (Kizumonogatari 1/2/3) into one entry.
  Index every id; never treat the scalar as canonical. AIOMetadata carries an
  explicit warning about this and it is an easy silent bug.
- `src/services/mal.js`: rating via Jikan (`api.jikan.moe/v4`, free, keyless,
  3 req/s + 60/min — respect it), disk-cached per title with a very long TTL.
  Kitsu as fallback where MAL has no entry.
- **Presence in the map is our anime detector**, replacing the current
  `original_language === 'ja' && genres includes Animation` heuristic. Better,
  and free.
- Degradation: if the file can't be fetched, keep the last good cache; with no
  cache at all, anime detection is simply off and everything takes the v5 LLM
  path. Degrades to today's behaviour, never to an open gate.

## Deliverable 2 — anime episode numbering

The outstanding risk from v5 Stage 1, and a plausible cause of Ciara's titles
not opening. The same map feeds it. Port AIOMetadata's
`oneToOneSeasonEpisodeFallback` approach: only resolve when there is exactly
one mapped IMDb season, its episode count matches TMDB's, and exactly one
episode matches — return null otherwise. Conservative by construction; a wrong
episode mapping is worse than none.

## Open decision — the `+1`

`judgementAge()` adds 1 because judging exactly at the limit was over-strict,
a workaround invented for anime. MAL's `pg_13` means literally "Teens 13 or
older", so it compares directly against `age_limit` with no fudge.

That splits the semantics: **anime judged on data at the true limit,
everything else judged by the model at +1.** Defensible — the fudge is
removed exactly where it was invented — but it is a real fork and it is
James's call, not one to make quietly. The alternative is keeping +1 on both
paths for consistency, at the cost of admitting anime one band higher than MAL
intends.

## Risks

- **MAL ratings are not ACB.** `pg_13` is a Japanese/US-facing label, not an
  Australian classification. It is a better signal than a model's recall, but
  it is not the standard we claim to judge against. Stage E remains the ACB
  opinion.
- **Brand-new titles are in neither source.** A 2026 premiere may have no MAL
  rating AND be past the model's cutoff, so it falls through to an LLM that
  vetoes it for being unknown. Same outcome as today — not a regression, but
  this stage does not fix it.
- **Jikan is an unofficial API.** Rate-limited and community-run. Long-lived
  caching keeps us light; a Jikan outage must degrade to "no rating known"
  (fall through to the LLM), never to dropping titles.
- **One more upstream dependency**, refreshed daily from GitHub. The ETag check
  makes the steady-state cost one HEAD request per day.

## Build order

1. `animeMap.js` — fetch, ETag refresh, slim indices, franchise arrays, tests
2. `mal.js` — rating lookup + long-lived cache; NSFW blacklist [C]
3. Wire [C] into lists, extras, search **and meta**; [D] into the age paths;
   pass MAL rating to [E] as evidence
4. Episode-numbering fallback (Deliverable 2)
5. Docs, version, tag `v5.2.0-beta`
